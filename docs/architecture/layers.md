# AI Content Engine — Architecture Layers

Данный документ описывает архитектурные слои системы AI Content Engine.
Система построена по принципу modular monolith с чётким разделением ответственности между слоями.

---

# 1. Infrastructure Layer

## Назначение

Обеспечивает техническую основу системы.

## Компоненты

- Docker (контейнеризация)
- PostgreSQL (структурированные данные)
- Qdrant (vector storage)
- MinIO (объектное хранилище)
- LLM backends (Local / API)

## Ответственность

- Хранение данных
- Хранение embeddings
- Хранение медиа
- Предоставление доступа к моделям

## Ограничения

- Не содержит бизнес-логики
- Не управляет workflow
- Не управляет контекстом

---

# 2. Kernel Layer

## Назначение

Центральный управляющий слой системы.

Kernel является единственной точкой координации между слоями.

## Компоненты

- LLM Router
- Task Manager
- Budget Controller
- Policy Engine
- Context Orchestrator

## Ответственность

- Создание и управление задачами (Task)
- Маршрутизация запросов к моделям
- Применение политик проекта
- Контроль бюджета
- Управление последовательностью выполнения

## Ограничения

- Не содержит логики конкретных агентов
- Не формирует prompts напрямую

---

# 3. Context Layer

## Назначение

Формирует актуальный контекст перед каждым вызовом агента.

## Типы памяти

- Static Memory (стратегия, стиль, правила)
- Dynamic Memory (недавние действия)
- Semantic Memory (vector-based retrieval)
- Summarized Memory (агрегированная история)

## Ответственность

- Сбор релевантных данных
- Ограничение размера контекста
- Предотвращение context drift
- Формирование структурированного пакета Context

## Принцип

Контекст формируется заново перед каждым вызовом.
Контекст не хранится внутри агента.

---

# 4. Agents Layer

## Назначение

Реализует конкретные роли в системе.

## Примеры агентов

- Analyzer
- Editor
- SEO Specialist
- Publisher
- Quality Control

## Свойства агента

- Имеет собственные prompts
- Может использовать RAG
- Получает Task + Context
- Возвращает структурированный AgentResult

## Ограничения

- Агент не знает о проекте напрямую
- Агент не управляет бюджетом
- Агент не вызывает другие агенты напрямую

---

# 5. Workflow Layer

## Назначение

Определяет порядок выполнения агентов.

## Ответственность

- Определяет последовательность шагов
- Управляет переходами
- Определяет необходимость Human approval
- Определяет policy исполнения

## Принцип

Workflow — это декларация процесса.
Workflow не содержит бизнес-логики агента.

---

# 6. Projects Layer

## Назначение

Изолированное runtime-окружение для контент-продукта.

## Проект определяет

- Активный workflow
- Набор агентов
- Бюджет
- Политику
- Ограничения
- Memory scope

## Принцип

Каждый проект изолирован от других.
Memory и бюджет не пересекаются между проектами.

---

# 7. Control & Visualization Layer

## Назначение

Предоставляет интерфейс мониторинга и управления системой.

## Функции

- Просмотр задач
- Статус агентов
- Контроль бюджета
- История действий
- Approval интерфейс

## Ограничения

- Не содержит логики выполнения
- Работает через Kernel API

---

# 8. Observability Layer

## Назначение

Обеспечивает наблюдаемость и контроль состояния системы.

## Включает

- Логирование
- Трассировку выполнения
- Метрики
- Учёт токенов
- Учёт стоимости
- Ошибки
- Audit trail

## Принцип

Каждый вызов модели и каждый шаг workflow логируется.

---

# 9. Versioning Layer

## Назначение

Контроль эволюции системы.

## Версионируются

- Prompts
- Workflows
- Agent configurations
- Project configurations

## Принцип

Каждое изменение сопровождается фиксацией версии.
Каждое изменение сопровождается документацией.

---

# 10. Human-in-the-Loop Layer

## Назначение

Обеспечивает контроль качества и управляемость.

## Функции

- Approval
- Manual edit
- Override
- Feedback
- Correction logging

## Принцип

Human Layer может вмешаться в любой шаг workflow.
Все вмешательства фиксируются и сохраняются.

---

# Общий принцип архитектуры

1. Агент не вызывается напрямую.
2. Контекст формируется перед каждым вызовом.
3. Kernel — единственная точка координации.
4. Все действия логируются.
5. Все изменения версионируются.
6. Любое расширение сопровождается документацией.
7. Код обязательно содержит комментарии.
